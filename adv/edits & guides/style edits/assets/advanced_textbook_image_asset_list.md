### Additional Placeholder Sizing Notes

| Placeholder Type | Intended Writing Volume | Typical Use Cases | Layout Guidance |
|---|---|---|---|
| PH-1 | 1–3 short sentences | planning notes, transformations, short reflections | should never visually dominate page |
| PH-2 | One developed paragraph | rewrites, editing tasks, guided drafting | maintain comfortable handwriting spacing |
| PH-3 | One complete short document | short emails, notices, explanations | should feel like a genuine drafting space |
| PH-4 | Extended freer-writing task | homework, full response tasks | may consume most of a page |
| PH-5 | Multi-stage portfolio work | capstone/project tasks | use segmented hierarchy, not one large box |

### Global Placeholder Layout Rules

- all placeholders should support handwritten responses
- avoid dense worksheet-style ruling
- preserve generous whitespace
- avoid heavy borders or shaded fills
- maintain strong compatibility with grayscale printing
- writing areas should feel professional and document-oriented rather than school-workbook oriented
- placeholder heights should expand proportionally with writing complexity
- avoid squeezing PH-3 and PH-4 into leftover page gaps
- PH-5 layouts should use strong visual segmentation and section labels

# 10. RECOMMENDED GENERATION METHODS

## 10.1 Overall Production Strategy

| Asset Category | Recommended Generation Method | Reason |
|---|---|---|
| Structural UI elements | Scripted pipeline (SVG/Python/HTML/CSS) | Maximum consistency and alignment |
| Placeholder systems | Scripted pipeline | Precision sizing and repeatability |
| Icons | Hybrid: AI master concepts → scripted cleanup | Ensures consistency while reducing drift |
| Diagrams/infographics | Scripted pipeline preferred | Easier version control and layout consistency |
| Editorial banners | Scripted pipeline | Typography/alignment must remain exact |
| Abstract module artwork | OpenAI image generation | More useful for atmospheric/editorial variation |
| Portfolio/capstone graphics | Hybrid workflow | Requires both structure and visual sophistication |

---

## 10.2 OpenAI API Image Generation Workflow

Recommended use cases:

- editorial module artwork
- abstract background graphics
- style-board generation
- icon concept exploration
- transformation-system visual concepts
- capstone visual identity concepts

Recommended implementation:

- use Codex in VS Code to build and run a local OpenAI Images API pipeline
- store the API key locally as an environment variable, never inside prompts, manifests, or committed files
- keep all prompts, settings, references, and outputs under version control except secrets
- use the OpenAI API only for assets where generative variation is useful
- use scripted SVG generation for assets that require exact repeatability

OpenAI API capabilities relevant to this project:

- GPT Image models can generate images from text prompts
- GPT Image models can use text plus image inputs, which is important for reference-image-driven generation
- image editing workflows can use existing reference assets to reduce design drift
- the API is suitable for programmable batch workflows and production asset pipelines

Recommended Codex-controlled workflow:

1. Read the asset manifest.
2. Identify each asset's generation method:
   - `scripted_svg`
   - `openai_image`
   - `hybrid`
3. For `scripted_svg`, generate the asset through local SVG templates.
4. For `openai_image`, call the OpenAI Images API using the locked prompt template.
5. For `hybrid`, generate an OpenAI concept image first, then recreate/clean the final version as SVG where possible.
6. Save all outputs to the correct asset folder.
7. Log prompt version, model, date, dimensions, source references, and revision notes.
8. Generate preview/contact sheets for human review.
9. Mark each asset as:
   - draft
   - approved
   - rejected
   - regenerate
   - needs manual cleanup

Security rule:

```text
Do not paste the API key into Codex prompts.
Do not store the API key in the repository.
Use an environment variable such as OPENAI_API_KEY.
Add .env files to .gitignore.
```

Recommended environment setup:

```powershell
setx OPENAI_API_KEY "your_api_key_here"
```

or use a local `.env` file that is excluded from Git:

```text
OPENAI_API_KEY=your_api_key_here
```

Recommended OpenAI API script starter prompt for Codex:

```text
Add OpenAI Images API support to the textbook asset-generation pipeline.

Requirements:
- read assets/manifests/assets.json
- select records with generation_method = "openai_image" or "hybrid"
- load OPENAI_API_KEY from the environment or local .env file
- never write the API key to logs
- combine the master style prompt with each asset-family prompt and asset-specific prompt
- support reference images from assets/references
- call the OpenAI Images API for image generation/editing
- save raw outputs to assets/openai-raw
- save selected variants to assets/openai-variants
- write metadata sidecar JSON files for every output
- include model, prompt template ID, full prompt text, reference images, dimensions, timestamp, and asset ID
- generate contact sheets for review
- update asset manifest statuses after generation
- include dry-run mode that prints planned API calls without calling the API
- include cost-control settings such as max_assets_per_run and max_variants_per_asset
- include retry/error handling
- include clear console progress output

Do not generate final SVG assets from OpenAI outputs automatically unless the asset is marked hybrid and a cleanup step is explicitly requested.
```

Recommended dry-run command concept:

```bash
python scripts/openai/generate_openai_assets.py --manifest assets/manifests/assets.json --dry-run
```

Recommended generation command concept:

```bash
python scripts/openai/generate_openai_assets.py --manifest assets/manifests/assets.json --family module_artwork --max-assets 3 --variants 2
```

Strong recommendation:

- run OpenAI generation in small batches
- approve one asset family before generating the next
- use reference images from approved outputs to control continuity
- treat OpenAI outputs as raw design candidates unless the asset is explicitly raster artwork
- convert final instructional UI assets to SVG wherever possible

## 13.7 Status Flow

| Status | Meaning | Allowed Next Status |
|---|---|---|
| `not_started` | Asset not generated yet | `draft` |
| `draft` | Generated but not reviewed | `approved`, `rejected`, `regenerate`, `needs_cleanup` |
| `needs_cleanup` | Concept accepted but requires SVG/manual cleanup | `draft`, `approved` |
| `regenerate` | Needs another generated version | `draft`, `rejected` |
| `rejected` | Do not use | `regenerate`, `not_started` |
| `approved` | Locked for production | no regeneration unless manually unlocked |
