# Part 1 — Replace hard-wired Lua style mapping

## 1. Add generic Lua filter

Create:

```text
textmaker/scripts/style_bridge.lua
```

Purpose:

Map Markdown fenced Div classes to DOCX paragraph styles using YAML metadata in the Markdown file.

Requirements:

1. Read `style_map` from Pandoc metadata.
2. For `FORMAT == "docx"` only, assign:

   ```lua
   el.attributes["custom-style"] = style_name
   ```

3. Leave unmapped Divs untouched.
4. Do not define fonts, colors, borders, spacing, or any visual properties.
5. Preserve Pandoc’s default handling for headings, lists, tables, normal paragraphs.
6. Support optional metadata:

   ```yaml
   style_bridge:
     remove_horizontal_rules: true
     preserve_div_line_breaks: true
   ```

7. If `remove_horizontal_rules` is true, remove standalone Markdown horizontal rules.
8. If `preserve_div_line_breaks` is true, convert soft breaks inside mapped Divs to line breaks.

Suggested implementation:

```lua
local stringify = pandoc.utils.stringify

local style_map = nil

local function get_meta_bool(section_name, key, default)
  local section = PANDOC_STATE.meta[section_name]
  if section and section[key] ~= nil then
    local value = stringify(section[key])
    return value == "true" or value == "1" or value == "yes"
  end
  return default
end

local function load_style_map()
  local map = {}
  local meta_map = PANDOC_STATE.meta["style_map"]

  if type(meta_map) ~= "table" then
    return map
  end

  for k, v in pairs(meta_map) do
    map[k] = stringify(v)
  end

  return map
end

local function preserve_line_breaks(el)
  if not get_meta_bool("style_bridge", "preserve_div_line_breaks", true) then
    return el
  end

  return pandoc.walk_block(el, {
    SoftBreak = function(_)
      return pandoc.LineBreak()
    end
  })
end

function HorizontalRule(el)
  if get_meta_bool("style_bridge", "remove_horizontal_rules", false) then
    return {}
  end
  return el
end

function Div(el)
  if FORMAT ~= "docx" then
    return el
  end

  if style_map == nil then
    style_map = load_style_map()
  end

  for _, class in ipairs(el.classes) do
    local style = style_map[class]
    if style and style ~= "" then
      el.attributes["custom-style"] = style
      return preserve_line_breaks(el)
    end
  end

  return el
end
```

# Part 7 — Critical XML rules for linked styles

## 7. Implement linked-style cloning correctly

When creating or updating a linked Div label style pair:

Paragraph style:

```xml
<w:style w:type="paragraph" w:customStyle="1" w:styleId="DivLabelLanguage">
  <w:name w:val="Div Label Language"/>
  <w:basedOn w:val="DivLabelBase"/>
  <w:next w:val="DivContent"/>
  <w:link w:val="DivLabelLanguageChar"/>
  ...
  <w:rPr>
    <w:color w:val="215868"/>
    <w14:textFill>
      <w14:solidFill>
        <w14:srgbClr w14:val="215868">
```

Character style:

```xml
<w:style w:type="character" w:customStyle="1" w:styleId="DivLabelLanguageChar">
  <w:name w:val="Div Label Language Char"/>
  <w:basedOn w:val="DivLabelBaseChar"/>
  <w:link w:val="DivLabelLanguage"/>
  ...
  <w:rPr>
    <w:color w:val="215868"/>
```

When changing color, update all three places:

```text
paragraph style: w:rPr/w:color/@w:val
paragraph style: w:rPr/w14:textFill/w14:solidFill/w14:srgbClr/@w14:val
character style: w:rPr/w:color/@w:val
```

If `w14:textFill` exists, keep it and update it. Do not leave old values.

If `w14:textFill` is missing in a cloned paragraph style, create it only if the source style has it. Prefer preserving the source structure.

```text
insert emoji run at start of label paragraph
apply existing label character style
normalize label text
set paragraph style to existing content/base style
```

But it must not:

```text
create styles
change style definitions
set hard-coded fonts/colors
invent fallback style names
```

The mapping from semantic class/style ID to label style must come from either:

```text
reference DOCX existing styles
YAML config
or explicit metadata
```

Not from hidden hard-coded fallback values except in tests.

## Phase D — Word COM finalization, explicit flag

These are useful but should remain optional:

```text
insert Quick Parts/building blocks
update fields
update TOC
repaginate
save through Word
```

Use flags such as:

```bash
--use-word-com
--update-fields
--insert-building-blocks
```

This avoids breaking Mac/Linux workflows.
