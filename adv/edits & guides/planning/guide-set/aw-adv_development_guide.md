# Advanced Development Guide
## *Administrative Writing, Advanced*

---

> **Purpose of this document**
> This file defines the authoritative planning-and-cross-check set for future advanced-book development work.
>
> Use this subfolder as the first stop when:
> - continuing development work
> - checking the live book against planning/control documents
> - deciding which source-of-truth document governs a question

This subfolder is the current development guide set:

- `adv/edits & guides/planning/guide-set`

---

## 1. What Belongs In This Guide Set

Only documents that actively govern ongoing development, review, or cross-checking belong here.

Current guide-set documents:

1. `aw-adv_project_todo_list.md`
   - project state, stage order, active next steps, key decisions

2. `aw-master_activity_menu.md`
   - activity-type definitions and pedagogical intent

3. `aw-adv_prompt-writing-policy.md`
   - prompt/task design standards

4. `aw-adv_unit_structural_profiles_revised.md`
   - planning source-of-truth for unit structure, Can-Do architecture, and planning notes

5. `aw-adv_n11_qa_checklist.md`
   - source-of-truth definition of the completed `N11` QA framework

6. `aw-adv_full-review_protocol.md`
   - source-of-truth definition of the post-`N11` full-review stage

---

## 2. How To Use The Guide Set

### 2.1 For current project status

Start with:

- `aw-adv_project_todo_list.md`

Use it to determine:

- the current active stage
- what is done
- what is deferred
- what the next project step is

### 2.2 For planning or revising live content

Cross-check in this order:

1. `aw-adv_unit_structural_profiles_revised.md`
2. `aw-master_activity_menu.md`
3. `aw-adv_prompt-writing-policy.md`

Use this sequence to answer:

- what activity belongs in a unit
- what that activity is supposed to do
- how its prompt and task design should be written

### 2.3 For QA and post-QA review work

Use:

1. `aw-adv_n11_qa_checklist.md` for the completed QA gate definition
2. `aw-adv_full-review_protocol.md` for the current editorial review stage

These documents define the standards for their stages.  
Their findings and round artifacts live outside this guide set.

---

## 3. Separation Of Functions

This guide set deliberately separates functions that had started to blur:

- `project status / sequencing`
  - `aw-adv_project_todo_list.md`

- `activity-type definition`
  - `aw-master_activity_menu.md`

- `prompt-design rules`
  - `aw-adv_prompt-writing-policy.md`

- `unit planning and cross-checking`
  - `aw-adv_unit_structural_profiles_revised.md`

- `QA stage definition`
  - `aw-adv_n11_qa_checklist.md`

- `full-review stage definition`
  - `aw-adv_full-review_protocol.md`

Do not duplicate these functions across multiple current documents unless there is a strong reason.

---

## 4. What Does Not Belong In This Guide Set

The following types of documents should not be treated as current planning authority:

- historical context summaries that no longer match current project state
- content-facing book material
- round-by-round findings files
- step prompts and working notes
- superseded drafts kept only for traceability

Those belong in supporting, reference, `n11`, `full-review`, or historical locations instead.

---

## 5. Supporting Material

Supporting but non-authoritative planning-adjacent files now live in:

- `adv/edits & guides/planning/supporting`

Use them for background only unless a current guide-set document explicitly points back to them.

---

## 6. Maintenance Rule

When a new document is created, decide immediately whether it is:

- a current guide-set control document
- a supporting background document
- a stage artifact
- or a historical note

Do not leave documents in the planning area without a clear function.
