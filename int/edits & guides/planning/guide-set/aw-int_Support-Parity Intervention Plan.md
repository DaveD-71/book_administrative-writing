# Intermediate Support-Parity Intervention Plan
## *Administrative Writing, Intermediate*

---

> **Purpose of this document**
> This file is the authoritative blocking control document for the intermediate book's support-parity intervention stage.
>
> It exists because the current intermediate book is not only simpler than the advanced book in level; it is also materially thinner in instructional support, quality-control architecture, and overall development depth.

---

## 1. Goal

The goal of this intervention is to bring the intermediate book up to the **same level of pedagogical support, development completeness, and editorial seriousness** as the advanced book while keeping:

- an A2-B1+ learner target
- a lower-load, more readable section structure
- shorter, clearer instructional wording than the advanced book

This intervention does **not** require importing the advanced `A-H` coded architecture directly.

It **does** require importing the same level of support depth:

- teachable model/example work
- explicit explanation of why stronger writing works
- visible review and self-edit support
- visible editing and revision support
- clear transfer and extension support
- sufficiently developed module volume

---

## 2. Why This Intervention Exists

The current intermediate book reads as **inferior**, not merely **simpler**.

The main causes are:

1. the planning layer allowed the book to remain underdeveloped in support depth
2. the first `INT-5` strengthening pass was too light to close the gap
3. the QA gate focused on local defects but did not treat support-parity failure as a whole-book blocker

As a result:

- the intermediate book often tells learners **what to do**
- it does not consistently explain **why the stronger version works**
- it lacks enough dedicated model-analysis, review, editing, and transfer architecture
- it remains far below advanced-level development volume

That combination makes the intermediate book feel less fully built than the advanced book.

---

## 3. Diagnosed Problems

### 3.1 Live-book support problems

- explicit teaching-rationale support is too thin across the book
- model/example comparison is inconsistent and often too light
- review / self-edit support is underdeveloped
- editing / revision support is not yet a visible universal unit function
- transfer / extension support is too light in many units
- the current module volumes are far below advanced-development parity

### 3.2 Control-layer problems

- `aw-int_activity_framework.md` protects readable structure, but does not yet enforce support-depth parity
- `aw-int_prompt-writing-policy.md` controls prompt shape, but not enough rhetorical explanation depth
- `aw-int_unit_structural_profiles.md` records goals, outputs, and exceptions, but not enough support-depth expectations
- `aw-int_qa_checklist.md` checks explicit teaching in principle, but does not hard-gate parity against the advanced book

### 3.3 QA-stage problem

`aw-int_qa_round1.md` currently records local `Revise` findings, but it does not yet record the larger book-level problem clearly enough:

- the intermediate book remains systemically underdeveloped relative to the required standard

That omission must be corrected before QA continues.

## 7. Implementation Rules For INT-5A

### 7.1 What must not happen

- do not copy the advanced `A-H` labels wholesale
- do not add volume by filler explanation
- do not make the intermediate book read like a compressed advanced clone
- do not let local QA fixes hide the larger parity failure

### 7.2 What must happen

- preserve intermediate readability
- increase development depth to advanced-equivalent seriousness
- make the book more teachable, not merely longer
- make support architecture visible in every module

### 7.3 Unit-level minimum support set

For every unit, the intervention rebuild must leave visible evidence of:

1. orientation to the target skill
2. model/example contrast or teachable before/after comparison
3. explicit explanation of the principle or decision logic
4. reader-effect or task-effect explanation (`why this works`)
5. guided support preparing freer production
6. review / self-edit support
7. editing / revision support
8. transfer / extension support

---

## 8. Definition Of Done For INT-5A

`INT-5A` is complete only when:

1. the guide-set control files explicitly encode support parity as a binding requirement
2. the six live module files have been rebuilt to the required support depth
3. each module falls within the required parity band or has an explicit approved exception recorded here
4. the active QA record clearly treats the underdevelopment problem as a book-level blocker
5. QA is ready to restart from the preliminary cross-check under the corrected control rules

Until all five conditions are met:

- `INT-5A` remains active
- `INT-6` remains blocked
