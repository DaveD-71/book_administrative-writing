## Module 1 - Foundations of Clear Administrative Writing

### Module Purpose

Module 1 establishes the foundation layer for the whole book:

- sentence clarity
- professional tone
- paragraph control

It prepares learners for the more explicitly email-based and document-based writing that begins in Module 2.

### Unit 1 - Clear Sentence Structure

**Unit learning goal**

Build control over short, clear administrative sentences using basic sentence patterns and simpler wording.

**Learner-facing Can-Do statements**

- I can write short clear sentences using basic subject-verb-object or subject-verb-complement patterns.
- I can rewrite long or indirect sentences in a simpler administrative style.
- I can write a short work-related message using clear sentence structure.

**Activity-purpose map**

- `Focus`: introduces sentence clarity as the first foundation skill
- `Warm-Up`: contrasts clearer and less clear sentence versions
- `Language 1`: teaches the basic sentence-pattern model
- `Practice - Identify the Parts`: builds sentence-pattern awareness
- `Practice - Build Clear Sentences`: moves from word groups to complete sentences
- `Language 2`: teaches simplification and removal of unnecessary complexity
- `Guided Practice`: applies clarity editing to administrative-style examples
- `Freer Practice`: produces a short, bounded work message
- `Reflection` / `Homework`: pushes transfer to workplace-like use

**Output-length expectation**

- guided output: sentence-level rewriting
- freer output: `3-4` sentence mini message
- homework output: `3-4` sentence transfer task

**Content flags for revision**

- this unit intentionally operates below full-document level and should remain the book's opening sentence-control unit
- later profile drafting should preserve a visible link from this sentence work into paragraph and email development
- duplicate example lines and dash inconsistency in the live source were corrected during the baseline audit

**Approved exception**

This unit is intentionally sentence-focused rather than paragraph- or document-focused because it functions as the lowest-level entry point for the book.

### Unit 2 - Professional Tone Basics

**Unit learning goal**

Introduce polite, professional email tone through request phrasing, neutral explanations, and short workplace email rewriting.

**Learner-facing Can-Do statements**

- I can make a short request sound more polite and professional.
- I can rewrite direct or emotional wording in a neutral workplace style.
- I can write a short professional email for a simple administrative situation.

**Activity-purpose map**

- `Focus`: frames professional tone as a practical email skill
- `Warm-Up`: contrasts direct and more professional sentence pairs
- `Language 1`: teaches standard polite request phrases
- `Language 2`: introduces neutral and respectful explanatory tone
- `Common Useful Phrases`: provides a reusable phrase bank for early email writing
- `Guided Practice`: rewrites an impolite draft email
- `Freer Practice`: produces a short professional email from a bounded scenario set
- `Reflection` / `Homework`: extends tone awareness to real workplace messages

**Output-length expectation**

- guided output: one rewritten short email
- freer output: `3-4` sentence professional email
- homework output: analysis and rewrite of a real email example

**Content flags for revision**

- the phrase-bank section is useful but sits outside the main `Language` numbering; decide later whether to retain that as a reference-bank exception or absorb it into a more standard section pattern
- keep the unit short and teachable rather than expanding it toward advanced-style realism

**Approved exception**

This unit uses a standalone phrase-bank section instead of a second fully parallel practice cycle; that is acceptable at this early stage if the later profile set records it intentionally.

### Unit 3 - Paragraph Structure

**Unit learning goal**

Teach learners how to build a clear paragraph with topic, support, and closing structure for administrative explanation and reporting.

**Learner-facing Can-Do statements**

- I can identify the main parts of a paragraph.
- I can organise sentences into a clear paragraph with logical flow.
- I can write a short administrative paragraph with topic, supporting detail, and closing.

**Activity-purpose map**

- `Focus`: frames paragraphing as a core administrative-writing skill
- `What Is a Paragraph?`: gives the core concept and boundaries
- `When Should You Start a New Paragraph?`: teaches paragraph-break logic
- `Warm-Up`: contrasts coherent and incoherent paragraphing
- `Language 1`: teaches the three-part paragraph model
- `Practice A`: identifies paragraph-part functions
- `Language 2`: teaches logical flow inside a paragraph
- `Practice B`: reorders and extends a paragraph sequence
- `Guided Practice`: rewrites a paragraph into a stronger structure
- `Freer Practice`: produces a bounded administrative paragraph
- `Reflection` / `Homework`: extends paragraph control to work-like material

**Output-length expectation**

- guided output: `5-6` sentence paragraph rewrite
- freer output: `5-7` sentence administrative paragraph
- homework output: one rewritten workplace paragraph

**Content flags for revision**

- this unit is more concept-heavy than Unit 1 or Unit 2 because it needs two explanatory sections before the warm-up
- later module planning should keep this as the bridge from sentence-level clarity to report-style and summary-style writing
- the practice-heading level and guided-practice typo were corrected during the baseline audit

**Approved exception**

This unit deliberately uses two front-loaded explanation sections before the main language/practice cycle because paragraph structure needs explicit concept teaching at this stage.

## Module 2 - Essential Email Writing for Administrative Contexts

### Module Purpose

Module 2 establishes the book's first full email-writing band:

- email layout and standard phrasing
- clarification writing
- delay/change communication
- polite request writing

It takes the sentence, tone, and paragraph foundations from Module 1 and applies them to short professional emails that are common in routine administrative work.

### Unit 4 - Email Layout and Standard Phrases

**Unit learning goal**

Build control over the basic architecture of a professional administrative email, including subject line, opening, body, closing, and sign-off.

**Learner-facing Can-Do statements**

- I can identify the main parts of a professional email.
- I can write a short, clear subject line for a workplace email.
- I can write a simple professional email with an appropriate opening and closing.

**Activity-purpose map**

- `Focus`: frames email structure as a practical workplace writing skill
- `What Does a Professional Email Look Like?`: introduces the core email-part model
- `Warm-Up`: contrasts a weak and stronger email to show why structure matters
- `Language 1`: teaches concise, professional subject lines
- `Practice - Improve the Subject Line`: shifts long sentence-like subjects into shorter email-style phrasing
- `Language 2`: introduces standard openings and closings
- `Practice - Choose an Opening and Closing`: matches openings/closings to reader type
- `Language 3`: teaches clear purpose-led opening sentences
- `Practice - Write the Opening Sentence`: turns situation prompts into direct opening lines
- `Guided Practice`: assembles a full short email from named parts
- `Freer Practice`: produces an independent full email from a bounded scenario set
- `Reflection` / `Homework`: pushes transfer from model structure into real-email rewriting

**Output-length expectation**

- guided output: one complete short email with a `2-3` sentence body
- freer output: one complete short email from a scenario choice
- homework output: one rewritten real email using the unit's structure

**Content flags for revision**

- this unit is the book's first full-email architecture unit and should remain clearly foundational rather than becoming a complex communication-skills unit
- the freer task currently has no explicit sentence or word band; later revision should decide whether to keep that open or align it more visibly with the surrounding early-book scaffolding
- later module planning should preserve the visible link from this layout work into the more function-specific email units that follow immediately after it

**Approved exception**

This unit intentionally spends more time on visible message structure and reusable phrase position than on deeper communicative nuance because it functions as the first explicit email-construction unit in the book.

### Unit 5 - Requesting Clarification

**Unit learning goal**

Teach learners how to ask focused, polite clarification questions when instructions, documents, or processes are unclear.

**Learner-facing Can-Do statements**

- I can write polite clarification questions about unclear or missing information.
- I can ask for specific details instead of writing a vague question.
- I can write a short clarification email about one clear problem.
- I can explain, in a short paragraph, why clarification is needed in a workplace situation.

**Activity-purpose map**

- `Focus`: frames clarification as a core administrative email function
- `Warm-Up`: identifies what information is missing or unclear in vague statements
- `Language 1`: introduces core clarification phrases
- `Practice A`: rewrites vague or abrupt questions as professional clarification questions
- `Language 2`: maps clarification writing to common workplace situations
- `Practice B`: improves weak clarification wording
- `Practice C`: generates clarification questions directly from bounded scenarios
- `Extended Guided Practice`: rewrites poor messages into clear clarification requests
- `Guided Practice`: writes a full clarification email about one defined gap
- `Freer Practice`: writes three focused short emails on separate clarification topics
- `Extended Writing Task`: converts the unit skill into a neutral explanatory paragraph about why clarification is needed
- `Reflection` / `Homework`: pushes transfer into a real unclear instruction or message

**Output-length expectation**

- guided output: one complete clarification email
- freer output: `3` short emails, each `3-5` sentences
- extended output: one `5-6` sentence informational paragraph
- homework output: one real-world clarification question based on recent instructions or email input

**Content flags for revision**

- this unit currently combines question-writing, email writing, and short explanatory-paragraph writing; later revision should check that those three output layers still feel cumulative rather than overloaded
- the `Reflection` section currently includes an additional mini email-writing task before the reflection questions; later revision should decide whether that belongs under reflection or should be reclassified as another production layer
- later profile drafting should preserve the unit's strong one-topic-per-email discipline, because that constraint helps keep clarification writing teachable at this level

**Approved exception**

This unit legitimately includes both multiple short clarification emails and a short summary paragraph because it bridges from question-level clarity into the wider reporting of unresolved information needs.

### Unit 6 - Communicating Delays or Changes

**Unit learning goal**

Teach learners how to explain delays, schedule changes, and procedural updates in a neutral, clear, and logically ordered way.

**Learner-facing Can-Do statements**

- I can explain a delay or change using neutral professional language.
- I can organise an update message in a clear order.
- I can write a short update email about a delay or schedule change.
- I can summarise a delay or change in a short report-style paragraph.

**Activity-purpose map**

- `Focus`: frames delay/change communication as a practical workplace-writing skill
- `Warm-Up`: identifies the information readers still need when a delay or change is announced
- `Language 1`: introduces neutral phrases for delays, changes, reasons, and next steps
- `Practice A`: rewrites underspecified delay/change messages into clearer updates
- `Language 2`: teaches the logic of update sequencing
- `Practice B`: builds ordering control for short update messages
- `Practice C`: expands short messages into clearer multi-sentence updates
- `Guided Practice`: writes a complete short update email around a defined delay
- `Freer Practice`: produces two short update emails from scenario choices
- `Extended Writing Task`: transfers the skill into a report-style summary paragraph
- `Reflection` / `Homework`: consolidates the neutral update language and applies it to a real notice or email

**Output-length expectation**

- guided output: one update email with a `2-3` sentence explanation block
- freer output: `2` short update emails, each `3-5` sentences
- extended output: one `5-7` sentence update-summary paragraph
- homework output: one rewritten delay/change notice or email

**Content flags for revision**

- the unit already teaches a useful message-order pattern and should remain the book's main delay/change logic unit rather than drifting into a general problem-reporting unit
- later revision should check whether the optional reason-language note is clear enough for learners to judge when explanation is needed and when it can stay minimal
- later module planning should preserve the bridge from email updates into the short report-style paragraph, because that bridge prepares later internal update writing

**Approved exception**

This unit legitimately extends beyond email-only writing because delay/change communication often appears both in short emails and in brief internal update paragraphs.

### Unit 7 - Making Requests Politely

**Unit learning goal**

Build control over polite administrative requests by teaching learners how to choose request forms, add necessary context, and soften tone without losing clarity.

**Learner-facing Can-Do statements**

- I can choose a polite request structure for a workplace email.
- I can make a request more specific and useful by adding the necessary detail.
- I can soften a request without making it unclear.
- I can write a short professional email requesting information or action.

**Activity-purpose map**

- `Focus`: frames polite request writing as a core administrative communication skill
- `Warm-Up`: contrasts direct and more professional requests
- `Language 1`: teaches the politeness scale for request forms
- `Practice A`: rewrites direct requests into polite, clearer versions
- `Language 2`: teaches how to make requests more specific
- `Practice B`: improves polite but vague requests
- `Language 3`: teaches softening without sounding weak
- `Practice C`: rewrites overly direct requests using softer phrasing
- `Guided Practice`: writes a complete request email with reason and softened closing
- `Freer Practice`: writes two short request emails from bounded options
- `Extended Writing Task`: transfers the unit skill into a neutral background paragraph for report context
- `Reflection` / `Homework`: consolidates preferred request structures and applies them to real workplace use

**Output-length expectation**

- guided output: one complete request email with a `1-2` sentence request block plus brief context
- freer output: `2` short request emails, each `3-5` sentences
- extended output: one `5-6` sentence request-background paragraph
- homework output: one real request rewritten using `2` different polite request structures

**Content flags for revision**

- this unit is appropriately dense because request writing needs explicit comparison of politeness levels, but later revision should keep the explanation readable and avoid turning the politeness scale into grammar-heavy theory
- the table formatting in `Language 1` should be checked during later module revision because it is more formatting-sensitive than the surrounding units
- later profile drafting should preserve the distinction between polite request language and the separate background-paragraph extension so the unit does not blur email action and report explanation into one uncontrolled task family

**Approved exception**

This unit intentionally uses an explicit politeness-scale table because comparing request forms side by side is a key part of the teaching load and would be harder to learn through examples alone.
