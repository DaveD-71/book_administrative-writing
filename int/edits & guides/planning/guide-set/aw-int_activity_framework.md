# Intermediate Activity Framework
## *Administrative Writing, Intermediate*

---

> **Purpose of this document**
> This file defines the structural logic of the intermediate book: what the main section types do, how units are expected to progress, where controlled variation is acceptable, and how later revision should avoid structural drift.
>
> Use it together with the structural profiles and prompt-writing policy when revising the live module files.

---

## 1. Core Principle

The intermediate book should feel simpler, more scaffolded, and more teachable than the advanced book.

That simplicity does **not** permit thinner pedagogy.

That means:

- the section pattern should be readable and predictable
- activity load should rise gradually
- explanations should be concrete and short
- guided support should do more visible preparation work than in the advanced book
- freer tasks should still feel like achievable workplace writing, not compressed advanced tasks
- the final instructional result should still be as fully developed as the advanced book in support depth and editorial seriousness

This framework now works with the hybrid `A-H` overlay defined in `aw-int_hybrid_ah_activity_standard.md`.  
The intermediate book keeps its learner-friendly section labels, but the live unit headings now also carry a visible `A-H` shell for cross-book alignment.

### 1.1 Support-parity rule

The intermediate book must retain lower-level readability **and** reach advanced-equivalent pedagogical completeness.

That means every unit must show functional equivalents of:

- teachable model/example work
- explicit explanation of why a stronger version works
- guided-to-freer transfer support
- review / self-edit support
- editing / revision support
- transfer / extension support

Do not accept a unit as complete merely because it is readable, scaffolded, and lower-level if it is still visibly thinner than the advanced book in instructional support.

## 2. Whole-Book Structural Model

The intermediate book has six modules with a `3 + 4 + 4 + 4 + 3 + 5` distribution:

1. **Module 1 - Foundations**
   - sentence clarity
   - tone basics
   - paragraph control

2. **Module 2 - Core workplace emails**
   - layout
   - clarification
   - changes and delays
   - polite requests

3. **Module 3 - Internal explanation and procedure**
   - procedures
   - notices
   - problem explanation
   - meeting summary

4. **Module 4 - External and semi-formal communication**
   - external inquiries
   - simpler explanation for outside readers
   - confusion-response tasks
   - consistent email style

5. **Module 5 - Editing and revision**
   - accuracy
   - clarity
   - tone control

6. **Module 6 - Integrated and capstone writing**
   - multi-audience tasks
   - revision plus notice writing
   - team consistency
   - multi-document communication
   - final portfolio / capstone preparation

Use this progression to judge whether a unit belongs where it currently sits and whether later complexity is prepared by earlier units.

## 3. Standard Intermediate Unit Spine

The default intermediate unit spine is:

1. `Focus`
2. concept or scenario setup
3. one or more `Language` sections
4. one or more short `Practice` sections
5. `Guided Practice`
6. `Freer Practice`
7. `Extended Writing Task` where the unit grows beyond a short bounded output
8. `Reflection`
9. `Homework`

Not every unit must show every label in exactly that form, but the learner should still feel a clear move from:

- understanding the target feature
- controlled use
- supported production
- more independent production

In addition, the unit must visibly include the support functions named in Section 1.1, even if they are integrated into the existing intermediate section labels rather than separated as advanced-style coded sections.

## 4. Section Function Expectations

### 4.1 Focus / opening frame

The opening section should:

- identify the unit's main writing skill
- frame it as a recognisable workplace need
- keep the load light enough for lower-level learners

It should not try to front-load the entire unit explanation.

### 4.2 Concept or scenario setup

This layer may appear as:

- `Warm-Up`
- `What Is...`
- a short contrastive example
- or another bounded setup section

Its job is to make the target feature visible before the main teaching begins.

It should also provide a teachable weak/strong contrast, model example, or another bounded comparison that the learner can later use in self-review.

### 4.3 Language sections

Language sections should:

- teach the principle behind the pattern
- provide short usable examples
- avoid long abstract explanation
- prepare the next practice layer directly

Intermediate language sections must explain enough for transfer, but not expand into advanced-style theory blocks.

They must also include at least one explicit explanation of why the stronger writing move works for the reader, purpose, tone, or document function.

### 4.4 Practice sections

Practice sections should move from:

- noticing
- identification
- controlled completion or rewriting
- short purposeful production

They should not repeat the same operation under different headings without a meaningful increase in demand.

### 4.5 Guided Practice

Guided Practice is the bridge between explanation and independent writing.

It should usually provide one or more of the following:

- a model or partial template
- supplied content to organise
- prompts that narrow the writing decision space
- a bounded short scenario

If Guided Practice is too open, the unit becomes harder than intended.

Guided Practice should also make the move into Freer Practice visible rather than leaving the learner to infer the transfer logic.

### 4.6 Freer Practice

Freer Practice should:

- preserve the unit's main target skill
- remove some support
- remain realistically achievable for A2-B1+ learners

It should feel like a genuine writing task, but still narrower than the advanced book's freer tasks.

Freer Practice should not be the first point in the unit where the learner is expected to discover why the target writing move works.

### 4.7 Extended Writing Task

Extended Writing Task is appropriate when the unit needs a more sustained output than the Freer Practice alone can provide.

Use it when:

- a full email or short document is the real target
- the learner has already been prepared through earlier layers
- the extra task extends the skill rather than duplicates the Freer Practice

Do not add an Extended Writing Task automatically to every unit.

Where homework alone is too light to carry real transfer, the unit should include a visible extension or transfer layer so the later support architecture does not collapse.

### 4.8 Reflection and Homework

Reflection should help the learner notice:

- what skill was practised
- what still feels difficult
- how the skill connects to real work

Homework should extend or transfer the unit skill.  
It should not introduce a new dominant skill family.

Reflection is not a substitute for review / self-edit support.  
Intermediate units still need a visible checking or revision function before the learner leaves the unit.

## 5. Approved Structural Variation

Variation is allowed, but it must be intentional and recorded in the structural profiles.

Approved variation types include:

- front-loaded concept teaching in early foundation units
- a phrase-bank or reference section where that genuinely supports beginners
- units with lighter output because they function as bridge units
- later units with integrated or multi-document tasks
- editing-focused units where the main output is revision rather than original drafting

Approved variation does **not** remove the support-parity requirement.

Unapproved variation includes:

- missing Guided Practice without a pedagogical reason
- repeated section labels that do the same job twice
- a sudden jump to complex multi-step writing without preparation
- unit-local structure that contradicts the book's readable instructional rhythm

## 6. Scenario And Output Rules

Intermediate tasks should use recognisable workplace situations:

- simple internal emails
- clarification requests
- delay or update messages
- notices
- short explanations
- meeting summaries
- procedure communication
- short multi-audience or portfolio tasks in later modules

Use scenario realism, but keep the context load controlled.  
Do not force learners to decode complex institutional detail before they can practise the writing skill.

Outputs should be:

- short and bounded early in the book
- gradually expanded across modules
- clearly prepared by the teaching and guided layers
- explicitly recorded in the structural profiles before normalization decisions are made

Sentence-band and paragraph-band expectations are acceptable in early and mid-book units where word counts would create unnecessary rigidity.

### 6.1 Volume-parity rule

Readable lower-intermediate prose does not justify severe underdevelopment in module volume.

During `INT-5A`, use the intervention-plan module bands as the control targets for development depth.  
Do not increase volume through filler. Increase it through real teaching, model analysis, review support, editing support, and transfer support.

---

## 7. Output-Control Policy

Before broad revision, preserve the current output behavior accurately.

That means:

- record actual output expectations in the structural profiles first
- identify where the book already uses short sentence tasks, paragraph tasks, email tasks, or short document tasks
- only then decide where normalization is useful

Do not normalize every unit toward the same output size.  
Intermediate progression depends on controlled variation, not flat uniformity.

---

## 8. Anti-Drift Rules

When revising the live module files:

- work from the module files only
- do not silently reintroduce authority from the older unit-file baseline
- keep the unit's dominant skill visible from opening to homework
- do not import advanced-style complexity just because the advanced system is more detailed
- record any intentional structural exception in the structural profiles

If a unit needs a major structural change, update:

1. `aw-int_unit_structural_profiles.md`
2. the live module file
3. any affected control note in the to-do list or review layer

---

## 9. How To Use This Framework

Use this file when:

- deciding whether a unit's current section pattern is acceptable
- checking whether a revised task sequence is still intermediate-appropriate
- testing whether a unit has enough preparation before freer writing
- judging whether later integrated tasks are adequately prepared

This file defines the allowed structural behavior of the intermediate book.  
The structural profiles define how each specific unit applies that behavior.
